## Context

目前 Compose TUI 在 `header composer view` 相關樣式中存在額外 bottom margin，以及 input 等元件的顏色覆寫。這些覆寫讓畫面與元件庫預設行為不一致，導致後續樣式調整需額外比對自訂值，增加維護負擔。本次變更聚焦於「回歸預設」，不更動互動流程與資料流。

## Goals / Non-Goals

**Goals:**
- 移除 `header composer view` 非必要的底部間距覆寫，使其使用預設值或等效 `0`。
- 移除 input 等元件的刻意色彩設定，回到元件庫預設外觀。
- 保持現有鍵盤操作、焦點切換與發信流程不變。

**Non-Goals:**
- 不重新設計 Compose TUI 版面結構。
- 不新增主題系統或動態換色機制。
- 不變更 sendmail、驗證或 CLI 參數行為。

## Decisions

1. **以刪除覆寫優先，必要時以 `0` 明確化間距**
   - 原因：若樣式系統允許繼承預設，刪除覆寫可降低未來衝突；若版面需要固定行為，則以 `0` 明確定義避免環境差異。
   - 替代方案：保留既有 margin 並僅微調數值。未採用，因仍是額外客製化。

2. **輸入元件回歸預設色彩，不再指定前景/背景色**
   - 原因：預設樣式通常已考量可讀性與狀態一致性，且可減少硬編碼色碼造成的主題衝突。
   - 替代方案：統一改成另一組固定色碼。未採用，因仍會增加後續維護成本。

3. **保留焦點可辨識性，但避免依賴輸入文字色彩**
   - 原因：焦點提示應主要由 panel 邊框與結構訊號承擔，避免以 input 色彩作為唯一辨識來源。
   - 替代方案：維持既有 input 顏色強調焦點。未採用，因違反「回歸預設」目標。

## Risks / Trade-offs

- [風險] 回歸預設後，某些終端主題下的對比可能與目前視覺習慣不同 → [緩解] 以現有焦點邊框與狀態列提示確保可辨識性，並在常見終端主題下手動驗證。
- [取捨] 少量視覺品牌感會降低 → [緩解] 保留必要的 panel 焦點框樣式，不影響操作辨識。
- [風險] 既有 UI 測試若綁定色值可能失敗 → [緩解] 更新測試斷言，改驗證行為與結構而非特定色碼。

## Migration Plan

1. 先調整 Compose 相關樣式定義，移除 bottom margin 與 input 色彩覆寫。
2. 執行測試與手動 TUI 檢視（重點：版面間距、焦點切換、輸入可讀性）。
3. 若發現主題相容性問題，僅補最小必要樣式，不回復全面硬編碼色彩。

## Open Questions

- `header composer view` 在現行程式碼中是否已由其他容器提供間距（若是，優先刪除本地 margin 而非設 `0`）？
- 是否存在除 input 之外的文字元件仍有硬編碼色彩，需一併納入本次清理？
